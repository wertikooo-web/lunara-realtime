## Что это за проект

Lunara Realtime — серверная платформа для голосового AI-устройства на ESP32-S3. Главная цель ближайшего этапа: стабильный физический прототип игрушки с быстрым живым разговором, затем демонстрационное видео и первые B2B-контакты с производителями.

## Текущий фактический стек

| Слой | Текущее решение |
|---|---|
| Железо | ESP32-S3 + внешний аудиокодек |
| Связь ESP32 ↔ сервер | WebSocket |
| Режим ввода | Push-to-talk |
| Сервер | Node.js 20+ |
| Realtime-провайдер | Google Gemini Live |
| SDK | `@google/genai` 2.11.x |
| Текущая модель | `gemini-3.1-flash-live-preview` |
| База | PostgreSQL через `pg` |
| Панель | Browser Lab + Parent Panel |
| Вход Gemini | PCM16LE mono, 16 кГц |
| Выход Gemini | PCM16LE mono, 24 кГц |
| Железный режим ESP32 | кодек/I2S 24 кГц на микрофон и динамик |

Старый стек Xiaozhi + Whisper + DeepSeek + Yandex SpeechKit больше не является текущей архитектурой этого репозитория. Не возвращать его без отдельного продуктового решения.

## Актуальная архитектура

```text
Микрофон через аудиокодек, PCM16LE mono 24 кГц
    → ESP32-S3
    → бинарные WebSocket-фреймы
    → Lunara Realtime Server
    → явный streaming resampler 24 → 16 кГц
    → Gemini Live
    → PCM16LE mono 24 кГц
    → сервер без изменения выходной частоты
    → ESP32-S3
    → аудиокодек и динамик 24 кГц
```

Ресемплинг входа должен быть явно интегрирован в штатный realtime pipeline. Запрещены preload через `node -r`, monkey-patching и скрытые bootstrap-механизмы.

## Протокол аудио

ESP32 в режиме 24/24 должна передавать при `session.start`:

```json
{
  "type": "session.start",
  "deviceId": "DEVICE_ID",
  "sampleRate": 24000
}
```

Правила:

- сервер читает `sampleRate` или `sample_rate`;
- разрешены только 16000 и 24000 Гц;
- 16000 проходит без преобразования;
- 24000 преобразуется сервером в 16000 до Gemini;
- выходные `audio.chunk` Gemini остаются 24000 Гц;
- если поле отсутствует, сервер использует обратную совместимость 16000 и обязан явно сообщить источник принятой частоты в `session.config.applied`;
- для ESP32, реально отправляющей 24000 Гц, поле `sampleRate: 24000` обязательно;
- неподдерживаемое значение отклоняется явной ошибкой;
- каноническая спецификация: `docs/ESP32_REALTIME_PROTOCOL.md`.

## MVP — ближайший фокус

1. Стабильный голосовой прототип ESP32 ↔ сервер ↔ Gemini Live.
2. Корректный звук без щелчков, обрезаний, повторов и зависаний.
3. Стабильный interrupt/barge-in.
4. Реальный тест на аудиокодеке и динамике игрушки.
5. Демонстрационное видео ребёнка с игрушкой.
6. Первые контакты с производителями игрушек.

Функции, которые не помогают получить прототип, демо, пользователя, производителя, партнёра или деньги, считаются secondary.

## Родительская панель и память

Сервер содержит Parent Panel, настройки устройства, профиль ребёнка, ограничения времени, аналитику и память в PostgreSQL.

Текущий `deviceId` является идентификатором, а не полноценным секретом. Перед внешним пилотом нужно добавить `deviceToken`, защищённую сессию родительской панели, позднее pairing code и связь пользователя с несколькими устройствами.

## Жёсткие правила работы с кодом

Всегда работать как senior software architect и senior developer с 20+ годами опыта.

1. Сначала изучить архитектуру, документацию, версии зависимостей и полный жизненный цикл данных.
2. Интегрировать изменения только в штатную архитектурную точку.
3. Не использовать monkey-patch, preload, скрытые bootstrap-модули, временные генераторы патчей и другую скрытую магию.
4. Не обходить ограничения GitHub, API или коннектора за счёт ухудшения архитектуры.
5. Если инструмент не позволяет безопасно изменить код, остановиться и запросить актуальный файл, ZIP, доступ или другой безопасный способ.
6. Не придумывать API, функции, параметры и совместимость библиотек по памяти. Проверять точную документацию и фактическую версию.
7. Предпочитать зрелые поддерживаемые библиотеки. Самописный код оставлять только с техническим обоснованием и усиленными тестами.
8. Одно изменение — одна задача. Без unrelated refactoring.
9. Обрабатывать start, normal flow, end, interrupt, error, timeout, reconnect, close, retry, reset и cleanup.
10. Не допускать опасных молчаливых значений по умолчанию. Любое fallback-поведение должно быть наблюдаемым и задокументированным.
11. Обновлять каноническую документацию, а не создавать противоречащие дополнительные документы.
12. Покрывать happy path, ошибки, границы, повторные вызовы, сброс состояния и регрессии.
13. Перед merge проверять синтаксис, тесты, diff, документацию и отсутствие побочных изменений.
14. Не сливать в `main` без тестов, senior-review и явного разрешения пользователя.
15. Честно указывать, что проверено автоматически, вручную, в Railway и на реальном железе.

## Что запрещено

- возвращать старый Xiaozhi/Whisper/DeepSeek/Yandex стек без явного решения;
- менять realtime-провайдера или модель без обсуждения;
- снижать выход Gemini с 24 до 16 кГц без отдельного решения;
- скрывать обработку аудио вне штатного pipeline;
- хранить детский голос без отдельного решения по приватности;
- добавлять сложную функцию до стабильного физического прототипа;
- объявлять smoke-тест доказательством production-готовности;
- сливать спорный код только потому, что он запускается;
- выбирать решение, удобное исполнителю, но неудобное проекту.

## Критерий хорошего результата

Другой senior-разработчик должен сразу видеть, где принимается аудио, где валидируется частота, где выполняется ресемплинг, где состояние создаётся и очищается, где аудио передаётся Gemini, какие тесты это проверяют и какая документация является канонической.
