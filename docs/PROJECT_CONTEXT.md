## Текущая архитектура

- ESP32-S3 с внешним аудиокодеком.
- Микрофон и динамик работают через один I2S с общей частотой.
- Выбран аппаратный режим 24 кГц на вход и выход кодека.
- ESP32 отправляет PCM16LE mono 24 кГц на Node.js realtime server через WebSocket.
- Сервер явно ресемплирует вход 24 → 16 кГц в штатном realtime pipeline.
- Gemini Live получает PCM16LE mono 16 кГц.
- Gemini Live возвращает PCM16LE mono 24 кГц.
- Сервер передаёт выход Gemini устройству без изменения частоты.
- Browser Lab остаётся отдельным тестовым клиентом и может использовать 16 кГц pass-through.

## Состояние входного ресемплинга

Архитектурное исправление завершено в `main`:

- удалён `inputResampleBootstrap.js`;
- удалён запуск через `node -r`;
- удалён monkey-patch `wsProtocol.createFrameParser`;
- добавлен явный модуль `src/realtime/inputAudioResampling.js`;
- `realtimeServer.js` напрямую валидирует частоту и вызывает ресемплер;
- 24000 Гц преобразуются в 16000 до `providerSession.sendAudio()`;
- 16000 Гц проходят byte-identical pass-through;
- replay-buffer хранит уже преобразованные байты, то есть именно тот поток, который был отправлен провайдеру;
- состояние сбрасывается на новый turn, interrupt, ошибку и закрытие;
- на `input_audio.end` выполняется `flush()` до завершения входа;
- выход Gemini 24000 Гц не изменяется;
- канонический `docs/ESP32_REALTIME_PROTOCOL.md` обновлён.

## Важное замечание review

Текущая реализация при полном отсутствии `sampleRate`/`sample_rate` не возвращает ошибку, а использует 16000 Гц для обратной совместимости. Это не молчаливое поведение: сервер логирует источник и сообщает клиенту `sample_rate_source: assumed_default_no_sample_rate` через `session.config.applied`.

Для реальной ESP32 в режиме 24/24 поле `sampleRate: 24000` обязательно. Перед внешним пилотом стоит отдельно решить, сохраняем ли legacy fallback или переводим протокол в строгий режим, где отсутствие частоты отклоняется ошибкой.

## Исследовательский backlog

Китайские realtime-провайдеры рассматриваются только как параллельный Provider №2 после стабилизации Gemini Live. До практической проверки аккаунта, API, русского/румынского speech output, latency, interrupt, data policy и фактического списания денег никаких архитектурных решений по переходу не принимать.

## Известные риски

- требуется проверка на реальной ESP32 и реальном аудиокодеке;
- legacy fallback при отсутствующем `sampleRate` может быть слишком мягким для production ESP32-протокола;
- нет завершённой device authentication для внешнего использования;
- нельзя параллельно менять одни и те же realtime-файлы несколькими агентами;
- смена realtime-провайдера без provider abstraction и измеримого A/B теста создаст лишний риск.
